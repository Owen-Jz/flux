Section 08

Motion & Animation

Motion in Flux serves to communicate state, provide feedback, and guide users through interactions. All animations respect system preferences and accessibility requirements.

Animation Principles

Flux animations are designed to be purposeful, subtle, and swift. Every motion should communicate something meaningful — whether it's confirming an action, indicating a state change, or guiding attention. We favor natural, physics-based movements that feel intuitive rather than mechanical.

Purposeful over decorative: Animation should never be purely cosmetic. Each transition serves a functional goal — reducing cognitive load, providing feedback, or establishing spatial relationships.

Easing Curves

Standard Easing
cubic-bezier(0.4, 0, 0.2, 1)

Default for most UI transitions. Fast start, gentle finish.

Spring Easing
cubic-bezier(0.34, 1.56, 0.64, 1)

Overshoots slightly, then settles. For confirmations and success states.

Reduced Motion Support

Accessibility: Respect User Preferences

All Flux animations automatically disable or reduce when users have prefers-reduced-motion enabled in their system preferences. This ensures accessibility for users with vestibular disorders or who are sensitive to motion.

@media (prefers-reduced-motion: reduce) { *, *::before, *::after { animation-duration: 0.01ms !important; animation-iteration-count: 1 !important; transition-duration: 0.01ms !important; } }

When reduced motion is preferred, elements snap instantly to their states without transitions. Focus indicators remain visible but skip their pulse animation.

Animation Do's and Don'ts

Do
  • Keep animations under 300ms for UI interactions
  • Use ease-out for entrances, ease-in for exits
  • Animate only elements that changed (not entire page)
  • Provide immediate feedback for touch/click
  • Match animation speed to perceived importance
  • Test animations on lower-end devices
Don't
  • Use animation purely for decoration
  • Exceed 500ms for single transitions
  • Chain multiple animations sequentially without purpose
  • Animate layout properties (width, height) on interactive elements
  • Use linear easing for anything but loading bars
  • Ignore the reduced motion preference
